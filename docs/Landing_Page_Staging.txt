# Setup Flow Build Plan

Goal: turn **Start a New Voyage** into a real setup flow without risking accidental save loss or breaking the current tracker.

Core target:

* Landing screen remains the app entry point.
* **Start a New Voyage** opens setup mode.
* Setup mode lets the DM configure the voyage.
* Existing saves are not overwritten until the DM clicks **Start Voyage** and confirms.
* The final setup values create a full tracker state, save it, publish the player-safe state, and enter tracker mode.

---

# Stage 1: Ship Name Foundation

Purpose: add the smallest new setup-related data field before building the setup form.

## Build

* Add `shipName` to the default tracker state.
* Default value: `The Marrowwind`.
* Add migration for old saves missing `shipName`.
* Add import validation for `shipName`.
* Include `shipName` in exported saves.
* Include `shipName` in published player state.
* Render `shipName` on the DM screen.
* Render `shipName` on the player screen.
* Add player-view fallback for old player snapshots missing `shipName`.

## Tests

* New default state has `shipName`.
* Old saves migrate with `shipName: "The Marrowwind"`.
* Invalid imported `shipName` is rejected or normalized according to current validation style.
* Published player state includes `shipName`.
* Player view renders `shipName`.

## Done When

* `npm run ci` passes.
* DM screen shows the ship name.
* Player screen shows the ship name.
* Old saves still load.

# Stage 3: Setup Validation and Start Voyage

Purpose: make the setup form create the real tracker state.

## Build

* Add setup form validation.
* Ship name rules:

  * required
  * trimmed
  * 1–60 characters
* Crew name rules:

  * required
  * trimmed
  * 1–30 characters
* Experienced sailor values must be boolean.
* Add `createTrackerStateFromSetup(setupDraft)`.
* Start Voyage behavior:

  * validate setup
  * show errors if invalid
  * if a saved voyage exists, confirm before overwrite
  * if confirmed, create tracker state from setup draft
  * save to localStorage
  * publish player state
  * enter tracker mode
  * log that a new voyage started

## Tests

* Empty ship name blocks Start Voyage.
* Too-long ship name blocks Start Voyage.
* Empty crew name blocks Start Voyage.
* Too-long crew name blocks Start Voyage.
* Valid setup creates a tracker state.
* Valid setup saves to localStorage.
* Valid setup publishes player state.
* Valid setup enters tracker mode.
* Existing save requires confirmation before replacement.
* Cancelled confirmation preserves the old save.
* Confirmed replacement overwrites the old save.
* Configured ship name appears in state.
* Configured crew names appear in state.
* Configured sailor/pirate crew selections are preserved.

## Done When

* `npm run ci` passes.
* Start Voyage works from setup.
* Invalid setup does not start.
* Existing saves cannot be lost accidentally.

* Return to landing or refresh.

* Click Back to Landing.

* Start setup again.
* Click Start Voyage.
* Cancel overwrite confirmation.
* Confirm old voyage is intact.
* Repeat and confirm overwrite.
* Confirm new voyage replaces old save.

# Stage 6: Future Expansion, Not Now

Do not build these during the first setup pass:

* Custom crew count
* Add/remove crew members
* Custom action lists
* Custom open sea event tables
* Custom scripted event tables
* Custom flooding thresholds
* Custom supply formulas
* Custom ship systems
* Full campaign/scenario builder

These should wait until the basic setup path is stable.

First setup target:

**Preset + ship name + six crew names + experienced sailor checkboxes.**
